# 布隆过滤器完整实现 —— 面试回答稿

---

## 面试官：你的布隆过滤器是怎么从头到尾设置的？

---

### 一、背景：为什么要用布隆过滤器？

我们项目是一个**高并发秒杀电商系统**（黑马点评升级版），核心场景是用户查询商铺和秒杀优惠券。

问题在于**缓存穿透**：如果有人恶意用不存在的 ID 疯狂请求，缓存里没有，每次请求都会穿透到数据库。如果量大，数据库扛不住。

解决方案就是布隆过滤器——它**判断一个元素"一定不存在"是 100% 准确的**。在查缓存之前先问布隆过滤器，如果返回 false，直接拒绝，根本不用查数据库。

布隆过滤器本质是一个**很长的二进制位数组 + 多个哈希函数**：
- `add(x)`：对 x 做 k 次哈希，把对应的 k 个 bit 位都置为 1
- `contains(x)`：对 x 做同样的 k 次哈希，如果所有 bit 位都是 1 → "可能存在"；如果有任意一位是 0 → "一定不存在"

---

### 二、技术选型：为什么用 Redisson？

我们底层用的是 **Redisson** 的 `RBloomFilter`，数据存在 Redis 里。选型原因：

1. **分布式共享**——布隆过滤器存在 Redis，所有服务实例共享同一份数据，不存在不一致问题
2. **持久化**——Redis 重启后数据不丢（RDB/AOF）
3. **开箱即用**——Redisson 封装了 bit 数组计算、哈希函数配置、`tryInit`、`add`、`contains` 等操作，不需要手写 Lua 脚本
4. **可配置**——只需告诉它"预期存多少元素"和"允许的误判率"，它自动算最优 bit 长度和哈希函数数量

配置一个 `RedissonClient` Bean，整个框架层共用：

```java
@Bean
public RedissonClient redissonClient(RedisProperties redisProperties, ...) {
    Config config = new Config();
    config.useSingleServer().setAddress("redis://127.0.0.1:6380");
    return Redisson.create(config);
}
```

### 三、配置设计：YAML 驱动，声明式创建

我们在 `application.yml` 中这样配置：

```yaml
bloom-filter:
  filters:
    shop:
      name: shop-id-bloom
      expected-insertions: 50000
      false-probability: 0.01
    voucher:
      name: voucher-id-bloom
      expected-insertions: 100000
      false-probability: 0.01
```

四个字段的含义：

| 字段 | 作用 | shop 的值 | voucher 的值 |
|------|------|-----------|-------------|
| `filters` 的 key (`shop` / `voucher`) | Map 的 key，也注册为 Bean 别名 | `shop` | `voucher` |
| `name` | Bean 正式名称 + Redis key 组成部分 | `shop-id-bloom` | `voucher-id-bloom` |
| `expected-insertions` | 预期插入元素数量，决定 bit 数组大小 | 50000 | 100000 |
| `false-probability` | 允许的误判率 | 0.01 (1%) | 0.01 (1%) |

**为什么两个过滤器值不一样？** 商铺和优惠券的 ID 数量不同。设得太大浪费内存，设得太小会导致实际误判率远超配置值。Redisson 根据这两个参数自动计算：
- shop (50000, 1%)：bit 数组约 **479,253 位 ≈ 58.5 KB**，**7 个哈希函数**
- voucher (100000, 1%)：bit 数组约 **958,506 位 ≈ 117 KB**，**7 个哈希函数**

如果想新增一个布隆过滤器（比如用户 ID），加一段配置就行，**零 Java 代码**。

### 五、自动装配：如何被 Spring 发现？

这是一个**自定义 framework 模块**（`dp-bloom-filter-framework`），它如何让 Spring Boot 自动加载？

通过 **Spring Boot 3.x 的 AutoConfiguration 机制**：

```
dp-bloom-filter-framework/src/main/resources/META-INF/spring/
    org.springframework.boot.autoconfigure.AutoConfiguration.imports
```

文件内容只有一行：
```
org.javaup.config.BloomFilterAutoConfiguration
```

Spring Boot 扫描 classpath 下所有 jar 的 `AutoConfiguration.imports` 文件，自动加载其中列出的配置类。业务模块只需引入 maven 依赖，**不需要 `@ComponentScan`，不需要 `@Import`**。

```java
@EnableConfigurationProperties(BloomFilterProperties.class)  // 激活属性绑定
public class BloomFilterAutoConfiguration {

    @Bean
    public BloomFilterHandlerFactory bloomFilterHandlerFactory() {
        return new BloomFilterHandlerFactory();  // 工厂 Bean，后面按名字取用
    }

    @Bean
    public BloomFilterHandlerRegistrar bloomFilterHandlerRegistrar(Environment environment) {
        return new BloomFilterHandlerRegistrar(environment);
    }
}
```

### 六、核心机制：动态注册 Bean（整个设计最精妙的地方）

你可能会想：为什么不直接在 `BloomFilterAutoConfiguration` 里用 `@Bean` 方法创建两个 `BloomFilterHandler`？

**因为 filter 的数量是配置驱动的。** 今天 2 个，明天可能 5 个、10 个。用 `@Bean` 方法写死数量是**不行的**。

我们用 `BeanDefinitionRegistryPostProcessor`——在 Spring 容器的**Bean 定义阶段**，动态读取配置，有多少个 filter 就注册多少个 Bean：

```java
public class BloomFilterHandlerRegistrar
        implements BeanDefinitionRegistryPostProcessor, PriorityOrdered {

    @Override
    public void postProcessBeanDefinitionRegistry(BeanDefinitionRegistry registry) {
        // 第一步：读取 bloom-filter.filters 下所有配置
        Binder binder = Binder.get(environment);
        Map<String, Filter> filters = binder.bind(
            "bloom-filter.filters",
            Bindable.mapOf(String.class, Filter.class)
        ).orElse(Collections.emptyMap());
        // 得到：{"shop"→Filter(...), "voucher"→Filter(...)}

        if (filters.isEmpty()) return;

        // 第二步：遍历每个配置，逐个注册 BeanDefinition
        filters.forEach((alias, cfg) -> {
            // alias = "shop"、"voucher"
            // cfg = Filter(name="shop-id-bloom", expectedInsertions=50000, ...)

            String beanName = cfg.getName();  // "shop-id-bloom"、"voucher-id-bloom"

            RootBeanDefinition bd = new RootBeanDefinition(BloomFilterHandler.class);

            // 构造器参数：RedissonClient, name, expectedInsertions, falseProbability
            bd.getConstructorArgumentValues().addIndexedArgumentValue(0,
                new RuntimeBeanReference("redissonClient"));  // 引用已存在的 Bean
            bd.getConstructorArgumentValues().addIndexedArgumentValue(1, beanName);
            bd.getConstructorArgumentValues().addIndexedArgumentValue(2, cfg.getExpectedInsertions());
            bd.getConstructorArgumentValues().addIndexedArgumentValue(3, cfg.getFalseProbability());

            // 注册到容器
            registry.registerBeanDefinition(beanName, bd);
            // 容器中现在有了 "shop-id-bloom" 这个 Bean

            // 额外注册别名：这样用 "shop" 也能拿到同一个 Bean
            if (!beanName.equals(alias)) {
                registry.registerAlias(beanName, alias);
            }
        });
    }

    @Override
    public int getOrder() {
        return Ordered.HIGHEST_PRECEDENCE;  // 最高优先级，最先执行
    }
}
```

**最终 Spring 容器中注册了：**

| Bean 名称 | 别名 | 类型 |
|-----------|------|------|
| `shop-id-bloom` | `shop` | `BloomFilterHandler` |
| `voucher-id-bloom` | `voucher` | `BloomFilterHandler` |

### 七、Bean 实例化：连接 Redis，创建布隆过滤器

每个 `BloomFilterHandler` 在构造时完成与 Redis 的交互：

```java
public class BloomFilterHandler {

    private final RBloomFilter<String> bloomFilter;  // Redisson 的布隆过滤器封装

    public BloomFilterHandler(RedissonClient redissonClient,
                              String name,              // "shop-id-bloom"
                              Long expectedInsertions,  // 50000
                              Double falseProbability)  // 0.01
    {
        // 1. 拼 Redis key：区分环境的 key
        //    SpringUtil.getPrefixDistinctionName() 读取 'prefix.distinction.name' 配置
        //    未配置时默认 "dp"，结果: "dp-shop-id-bloom"
        String redisKey = SpringUtil.getPrefixDistinctionName() + "-" + name;

        // 2. 从 Redisson 获取布隆过滤器代理对象
        RBloomFilter<String> bf = redissonClient.getBloomFilter(redisKey);

        // 3. 初始化：tryInit 是幂等的——如果 Redis 中该 key 已存在则不覆盖
        bf.tryInit(expectedInsertions, falseProbability);
        // → Redis 执行: BF.RESERVE dp-shop-id-bloom 0.01 50000

        this.bloomFilter = bf;
    }

    // 添加元素
    public boolean add(String data) {
        return bloomFilter.add(data);   // Redis 执行: BF.ADD dp-shop-id-bloom "12345"
    }

    // 判断是否可能存在
    public boolean contains(String data) {
        return bloomFilter.contains(data);  // Redis 执行: BF.EXISTS dp-shop-id-bloom "12345"
    }
}
```

**关键设计点**：

1. **环境隔离**：Redis key 拼了环境前缀。正式环境配 `prefix.distinction.name=prod`，key 就是 `prod-shop-id-bloom`，开发环境是 `dev-shop-id-bloom`。多套环境共用同一 Redis 也不会冲突。

2. **`tryInit` 幂等**：配置 `expected-insertions: 50000`，但如果 Redis 中已经用 `expected-insertions: 30000` 初始化过了，`tryInit` 不会覆盖。这是合理的——布隆过滤器依赖初始化参数计算 bit 数组，覆盖会导致数据不一致。

### 八、启动时全量初始化：把存量数据灌进去

布隆过滤器刚创建时是空的，启动时必须把数据库已有的数据全部加载进去，不然已有的 ID 也会被拦截：

```java
@Slf4j
@Order(1)          // 最高优先级，确保在所有组件初始化后、接受请求前执行
@Component
public class BloomFilterDataInit {

    @Resource
    private IShopService shopService;
    @Resource
    private ISeckillVoucherService seckillVoucherService;
    @Resource
    private BloomFilterHandlerFactory bloomFilterHandlerFactory;

    @PostConstruct
    public void init() {
        // 全量加载商铺 ID
        log.info("==========初始化商铺的布隆过滤器==========");
        List<Shop> shopList = shopService.list();           // SELECT * FROM tb_shop
        for (Shop shop : shopList) {
            bloomFilterHandlerFactory
                .get("shop-id-bloom")                       // 拿到商铺对应的 Handler
                .add(String.valueOf(shop.getId()));         // 把每个 ID 加入布隆过滤器
        }

        // 全量加载秒杀优惠券 ID
        log.info("==========初始化优惠券的布隆过滤器==========");
        List<SeckillVoucher> seckillVoucherlist = seckillVoucherService.list();
        for (SeckillVoucher seckillVoucher : seckillVoucherlist) {
            bloomFilterHandlerFactory
                .get("voucher-id-bloom")
                .add(String.valueOf(seckillVoucher.getVoucherId()));
        }
    }
}
```

**面试官可能会追问：** 数据量大怎么办？全量查表不慢吗？
- 我们当前数据量在万级别，`list()` 全量加载可以接受
- 如果到百万级别，可以改成**分页加载 + 批量 add**，或者用 **Redis 的布隆过滤器持久化（BF.SCANDUMP / BF.LOADCHUNK）** 做热重启，不用每次从 DB 重新灌

### 九、运行时使用

#### 工厂模式获取

```java
@Component
public class BloomFilterHandlerFactory implements ApplicationContextAware {
    private ApplicationContext applicationContext;

    public BloomFilterHandler get(String name) {
        return applicationContext.getBean(name, BloomFilterHandler.class);
    }
}
```

业务代码注入工厂，按名字获取对应的 Handler。

#### 场景一：查询时防缓存穿透（最关键）

以商铺查询为例：

```java
@Service
public class ShopServiceImpl implements IShopService {

    @Resource
    private BloomFilterHandlerFactory bloomFilterHandlerFactory;

    public Shop queryById(Long id) {
        // 1. 先查 Redis 缓存
        String key = RedisConstants.CACHE_SHOP_KEY + id;
        Shop shop = redisCache.get(key, Shop.class);
        if (shop != null) {
            return shop;   // 缓存命中，直接返回
        }

        // 2. 缓存未命中 → 问布隆过滤器
        if (!bloomFilterHandlerFactory
                .get("shop-id-bloom")
                .contains(String.valueOf(id))) {
            // contains 返回 false → 100% 确定这个 ID 不存在
            throw new RuntimeException("查询商铺不存在");  // 直接拦截，不查 DB
        }

        // 3. contains 返回 true → "可能存在"（1% 的误判）
        //    需要查 DB 最终确认
        shop = getById(id);
        if (shop != null) {
            redisCache.set(key, shop, ...);  // 回写缓存
        }
        return shop;
    }
}
```

**关键逻辑**：
- 布隆过滤器说"不存在" → **100% 确定**，直接拒绝请求
- 布隆过滤器说"可能存在" → 有 1% 的概率误判（实际上不存在），再查 DB 确认

如果没有布隆过滤器，恶意用户用不存在 ID 狂刷，**每次都会穿透到 DB**。有了布隆过滤器，99% 的无效请求在 Redis 层就被拦截了。

#### 场景二：新增数据时同步写入

```java
public Result saveShop(Shop shop) {
    shop.setId(snowflakeIdGenerator.nextId());
    save(shop);       // 持久化到 MySQL

    // 同步写入布隆过滤器
    bloomFilterHandlerFactory
        .get("shop-id-bloom")
        .add(String.valueOf(shop.getId()));

    return Result.ok(shop.getId());
}
```

这样后续查询这个新 ID 时，布隆过滤器就会返回 true。

#### 场景三：库存加载前校验

```java
@ServiceLock(lockType = LockType.Read, name = UPDATE_SECKILL_VOUCHER_STOCK_LOCK, ...)
public void loadVoucherStock(Long voucherId) {

        throw new RuntimeException("查询秒杀优惠券不存在");
    }
    // ... 执行库存加载逻辑
}
```

### 十一、面试常见追问

#### Q1：布隆过滤器不能删除元素，你们怎么处理数据被删除的情况？

**答**：我们的场景是商铺 ID 和优惠券 ID，属于**只增不减**的数据。商铺和优惠券不会物理删除，只会逻辑下架。所以布隆过滤器只增不删的特性正好匹配。

如果未来需要支持删除，可以考虑**计数布隆过滤器（Counting Bloom Filter）**，Redisson 目前不原生支持，需要自己实现。

#### Q2：1% 的误判率可以接受吗？有没有优化？

**答**：1% 已经足够。布隆过滤器拦截后还有缓存和 DB 两层兜底：
- 布隆过滤器 → 缓存 → DB
- 即使 1% 的无效请求穿透布隆过滤器，缓存也不会命中，最后走 DB 查不到返回空，不会返回错误数据

如果对误判率敏感，可以调成 `0.001`（0.1%），代价是 bit 数组更大、内存更多。

#### Q3：Redis 挂了怎么办？布隆过滤器数据丢了怎么办？

**答**：
1. Redis 有 RDB/AOF 持久化，重启后数据恢复
2. 即使 Redis 彻底丢失数据，`BloomFilterDataInit` 在应用重启时会从 DB 全量重建
3. 重建期间布隆过滤器不完整，最坏情况是部分已有 ID 被判定为"不存在"导致查询失败——好在重建很快（万级数据量秒级完成）

#### Q4：你们是怎么做到动态增加布隆过滤器而不用改代码的？

**答**：整个注册流程是**配置驱动**的。`BloomFilterHandlerRegistrar` 实现了 `BeanDefinitionRegistryPostProcessor`，在 Bean 定义阶段动态读取 YAML 配置。新增一个过滤器只需要在 `application.yml` 加一段配置，框架会自动：
1. 绑定属性
2. 注册 `BloomFilterHandler` Bean
3. 在 Redis 中创建布隆过滤器
4. 不需要改任何 Java 代码，不需要重启业务逻辑

#### Q5：为什么用 `BeanDefinitionRegistryPostProcessor` 而不是 FactoryBean 或 @Bean？

**答**：
- `FactoryBean`：需要为每种布隆过滤器写一个类，做不到纯配置驱动
- `@Bean` 方法：filter 数量写死在代码里，新增 filter 要改配置类
- 而 `BeanDefinitionRegistryPostProcessor` 在**容器刷新的 Bean 定义阶段**执行，可以读取任意数量的配置项，动态创建任意数量的 BeanDefinition。真正做到了"配置即代码"
